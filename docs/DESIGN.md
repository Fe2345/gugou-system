# 谷购系统 Django 后端设计文档

## 1. 文档目的

本文档用于指导谷购二次元谷子资产管理与交易平台的后端开发，明确 Django 后端的总体架构、模块划分、核心数据模型、模块依赖关系和开发边界，便于后续团队成员或 Agent 按模块协同开发。

本系统后端设计依据现有系统设计报告和课程设计方法展开。系统设计报告已经明确系统数据库需覆盖用户、商品、资产、订单、支付、换物、拼团、价格记录、信用评价和系统日志等核心数据，并要求保证业务数据的完整性、可追踪性和一致性。 同时，系统规划报告根据 U/C 矩阵将系统划分为基础数据管理、核心业务处理、决策分析、系统支撑与运维四类子系统，这一划分可直接作为后端模块拆分依据。

## 2. 总体架构设计

后端采用 Django + Django REST Framework 作为主体框架，数据库采用 MySQL 或 MariaDB，Redis 用于缓存、验证码、登录状态辅助、交易锁定和高频状态控制。

系统采用 B/S 架构，前端通过 HTTP API 调用后端服务。系统设计报告中也明确谷购系统采用浏览器/服务器结构，客户端包括普通用户端和管理端，应用服务器负责用户管理、商品管理、资产管理、交易支付、换物管理、拼团管理、价格分析、信用评价和后台审核等业务处理。

后端内部建议采用以下分层：

| 层级    | 职责                            |
| ----- | ----------------------------- |
| 接口层   | 接收前端请求，完成身份校验、权限校验、参数入口控制     |
| 序列化层  | 完成请求参数校验、响应数据格式化              |
| 业务服务层 | 承担订单生成、资产转移、换物完成、拼团成功等核心业务逻辑  |
| 数据模型层 | 定义 Django Model，对应数据库表和核心业务对象 |
| 公共支撑层 | 提供统一响应、异常处理、分页、日志、编号生成、权限控制   |
| 外部接口层 | 对接第三方支付、消息通知等外部服务             |

第八讲 PPT 中强调，软件体系结构模型需要说明系统功能、涉及哪些类、类之间如何联系、对象如何协作，并采用通用接口界面层、系统业务对象层、系统数据库层来描述软件逻辑结构。 因此，后端不应把业务逻辑全部堆在 View 中，而应形成接口层、业务层、模型层的清晰分工。

## 3. Django 项目结构建议

```text
gugou_backend
  config
    settings.py
    urls.py
    asgi.py
    wsgi.py

  apps
    common
    accounts
    products
    assets
    market
    orders
    exchanges
    teams
    pricing
    credits
    operations
```

各模块说明如下：

| Django App | 模块名称    | 主要职责                        |
| ---------- | ------- | --------------------------- |
| common     | 公共基础模块  | 统一响应、异常处理、分页、编号生成、基础模型、通用权限 |
| accounts   | 用户与权限模块 | 注册、登录、用户资料、role 权限、账户状态     |
| products   | 商品档案模块  | 谷子商品库、IP、角色、品类、稀有度、参考价格     |
| assets     | 用户资产模块  | 用户持有资产、资产数量、入手价、当前估值、资产状态   |
| market     | 交易市场模块  | 用户发布出售挂单、商品上架、交易市场列表        |
| orders     | 订单支付模块  | 下单、支付记录、订单状态流转、交易完成         |
| exchanges  | 换物模块    | 换物请求、换物匹配、双方确认、资产互换         |
| teams      | 拼团模块    | 拼团项目、拼团参与记录、拼团状态控制          |
| pricing    | 价格与估值模块 | 价格记录、价格走势、资产估值更新            |
| credits    | 信用评价模块  | 信用分变动、信用评价记录、风控参考           |
| operations | 后台与日志模块 | 管理端统计、系统日志、管理员操作审计          |

这里不再单独建立普通用户系统和管理员系统。所有用户统一进入 accounts 模块，通过 User.role 区分普通用户和管理员。管理员相关功能通过权限控制进入后台接口，而不是另建一套用户表和登录体系。

## 4. 用户与权限设计

### 4.1 统一用户模型

原设计报告中有用户表和管理员表。考虑到后端实现和权限维护，建议合并为统一 User 模型，通过 role 字段区分身份。报告中的管理员表可转化为用户模型中的管理员角色，或通过 AdminProfile 作为扩展资料，不再独立承担登录认证。

### 4.2 User 核心字段

| 字段           | 含义      |
| ------------ | ------- |
| user_id      | 用户编号，主键 |
| phone        | 手机号，唯一  |
| nickname     | 昵称      |
| password     | 加密密码    |
| role         | 用户角色    |
| credit_score | 信用分     |
| status       | 账户状态    |
| created_at   | 注册时间    |
| updated_at   | 更新时间    |

role 建议取值：

| role  | 含义   |
| ----- | ---- |
| user  | 普通用户 |
| admin | 管理员  |

status 建议取值：

| status   | 含义  |
| -------- | --- |
| normal   | 正常  |
| frozen   | 冻结  |
| disabled | 停用  |
| deleted  | 已注销 |

### 4.3 权限规则

| 操作类型    | 普通用户 | 管理员      |
| ------- | ---- | -------- |
| 浏览商品和价格 | 允许   | 允许       |
| 管理个人资产  | 仅限本人 | 可查看和监管   |
| 发布出售挂单  | 允许   | 可监管      |
| 创建订单    | 允许   | 可查看和处理异常 |
| 发起换物    | 允许   | 可监管      |
| 发起或参与拼团 | 允许   | 可监管      |
| 商品审核与冻结 | 不允许  | 允许       |
| 用户冻结与解冻 | 不允许  | 允许       |
| 查看系统日志  | 不允许  | 允许       |

## 5.3 assets 用户资产模块

### 模块职责

assets 模块负责管理用户实际持有的谷子资产。资产必须关联用户和商品，用于资产展示、资产估值、出售、换物和交易锁定。

系统设计报告中明确用户资产表关联用户表和商品表，用于资产管理、估值计算和换物请求撮合。

### 核心模型

| 模型                     | 说明        |
| ---------------------- | --------- |
| UserAsset              | 用户资产      |
| AssetFlow              | 资产流转记录    |
| AssetValuationSnapshot | 资产估值快照，可选 |

### UserAsset 核心字段

| 字段            | 含义      |
| ------------- | ------- |
| asset_id      | 资产编号，主键 |
| owner         | 所属用户    |
| product       | 关联商品    |
| quantity      | 持有数量    |
| acquire_price | 入手价格    |
| acquired_at   | 入手时间    |
| current_value | 当前估值    |
| status        | 持有状态    |
| source        | 来源      |
| created_at    | 创建时间    |

### 资产状态

| 状态         | 含义   |
| ---------- | ---- |
| holding    | 持有中  |
| selling    | 出售中  |
| exchanging | 换物中  |
| locked     | 交易锁定 |
| sold       | 已售出  |
| exchanged  | 已换出  |
| invalid    | 已失效  |

### 需要修正的编号问题

原报告中资产 id 设计为 `AS + 用户ID`，但一个用户可以拥有多条资产记录，因此该编码会导致同一用户多个资产主键冲突。建议改为：

```text
AS + 年月日时分秒 + 四位随机数
```

或：

```text
AS + 年月日 + 用户后四位 + 四位流水号
```

## 5.5 orders 订单与支付模块

### 模块职责

orders 模块负责下单、订单状态流转、支付记录生成、支付结果处理和交易完成。支付记录需要保证资金流可追溯，系统设计报告中也明确支付记录表用于记录支付流水、支付状态和第三方交易流水号。

### 核心模型

| 模型             | 说明       |
| -------------- | -------- |
| Order          | 订单       |
| PaymentRecord  | 支付记录     |
| OrderStatusLog | 订单状态变更记录 |

### Order 核心字段

| 字段           | 含义   |
| ------------ | ---- |
| order_id     | 订单编号 |
| buyer        | 买家   |
| seller       | 卖家   |
| listing      | 关联挂单 |
| product      | 商品   |
| quantity     | 数量   |
| amount       | 订单金额 |
| status       | 订单状态 |
| created_at   | 下单时间 |
| paid_at      | 支付时间 |
| completed_at | 完成时间 |

### PaymentRecord 核心字段

| 字段             | 含义       |
| -------------- | -------- |
| payment_id     | 支付流水编号   |
| order          | 关联订单     |
| payer          | 支付用户     |
| amount         | 支付金额     |
| pay_method     | 支付方式     |
| status         | 支付状态     |
| third_trade_no | 第三方交易流水号 |
| created_at     | 创建时间     |

### 订单状态

| 状态              | 含义  |
| --------------- | --- |
| created         | 已创建 |
| pending_payment | 待支付 |
| paid            | 已支付 |
| completed       | 已完成 |
| cancelled       | 已取消 |
| closed          | 已关闭 |
| refunded        | 已退款 |

### 订单核心规则

创建订单时，系统需要锁定 Listing 和对应 UserAsset，防止同一资产被重复购买。

支付成功后，系统需要同时完成以下操作：

| 操作     | 说明                 |
| ------ | ------------------ |
| 更新支付记录 | 标记支付成功             |
| 更新订单状态 | 从待支付变为已支付或已完成      |
| 转移资产归属 | 资产从卖家转移给买家         |
| 更新挂单状态 | 从 locked 变为 sold   |
| 写入价格记录 | 将成交价写入 PriceRecord |
| 写入信用记录 | 根据交易结果更新信用数据       |
| 写入系统日志 | 记录交易操作             |

以上操作必须放在数据库事务中执行。

## 5.6 exchanges 换物模块

### 模块职责

exchanges 模块负责用户发布换物请求、其他用户提交匹配资产、发起人确认换物和双方资产互换。

系统设计报告中明确换物请求表关联用户资产表和用户表，用于撮合用户间资产交换，并通过状态字段跟踪请求是否完成或取消。

### 核心模型

| 模型                | 说明       |
| ----------------- | -------- |
| ExchangeRequest   | 换物请求     |

### ExchangeRequest 核心字段

| 字段                    | 含义     |
| --------------------- | ------ |
| exchange_id           | 换物请求编号 |
| owner                 | 发起用户   |
| offered_asset         | 发起方资产  |
| target_condition      | 目标条件   |
| price_difference_note | 补差说明   |
| status                | 请求状态   |
| created_at            | 创建时间   |

### ExchangeMatch 核心字段

| 字段              | 含义     |
| --------------- | ------ |
| match_id        | 匹配编号   |
| request         | 关联换物请求 |
| applicant       | 申请用户   |
| applicant_asset | 申请方资产  |
| status          | 匹配状态   |
| created_at      | 创建时间   |

### 换物状态

| 状态        | 含义  |
| --------- | --- |
| active    | 发布中 |
| matched   | 已匹配 |
| completed | 已完成 |
| cancelled | 已取消 |
| expired   | 已过期 |

### 设计补充说明

原报告中的换物请求表只能表达“用户想换什么”，但不能完整表达“谁拿什么来换、是否被接受、是否被拒绝”。因此需要增加 ExchangeMatch。这个模型能够支持多个用户对同一换物请求提交不同资产，也方便管理端审查和状态追踪。

### TeamParticipant 核心字段

| 字段             | 含义     |
| -------------- | ------ |

系统设计报告中明确信用评价记录表用于保存用户交易后的信用变化记录，支撑信用评分体系与风控策略。

| 字段               | 含义     |
| ---------------- | ------ |
| credit_record_id | 信用记录编号 |
| user             | 关联用户   |

| reason           | 变动原因   |
| related_order    | 关联订单   |
| created_at       | 创建时间   |

### 信用设计规则

信用分不能只修改 User.credit_score。每次信用变化必须写入 CreditRecord，然后同步更新 User.credit_score。这样可以保证信用变化可追溯。

## 7. 模块依赖关系

| 模块         | 依赖模块                                                 | 说明                |
| ---------- | ---------------------------------------------------- | ----------------- |
| products   | accounts                                             | 商品创建和审核需要记录操作人    |
| assets     | accounts, products                                   | 资产必须属于用户并关联商品     |
| market     | accounts, assets, products                           | 挂单由用户基于资产发布       |
| orders     | accounts, market, products, assets, pricing, credits | 订单会影响资产、价格和信用     |
| exchanges  | accounts, assets                                     | 换物围绕双方资产展开        |
| teams      | accounts, products                                   | 拼团由用户发起并围绕商品展开    |
| pricing    | products, orders, assets                             | 价格来自成交记录并反向影响资产估值 |
| credits    | accounts, orders, exchanges                          | 信用变化来自交易和换物行为     |
| operations | 全部业务模块                                               | 记录操作日志和后台审计       |

整体依赖方向应保持单向，基础模块不反向依赖业务模块。也就是说，accounts 和 products 不应依赖 orders，orders 可以依赖 accounts 和 products。

## 8. 关键业务流程设计

### 8.1 出售交易流程

```text
用户选择资产
创建出售挂单
资产状态变为 selling
买家创建订单
挂单状态变为 locked
资产状态变为 locked
买家支付
支付记录变为 success
订单变为 completed
资产归属转移给买家
挂单变为 sold
写入价格记录、信用记录和系统日志
```

### 8.2 换物流程

```text
用户选择资产发起换物
资产状态变为 exchanging
其他用户提交匹配资产
发起人接受匹配
双方资产锁定
系统交换资产归属
换物请求变为 completed
写入资产流转记录、信用记录和系统日志
```

### 8.3 拼团流程

```text
用户创建拼团项目
其他用户参与拼团
系统增加 current_count
达到 target_count 后拼团成功
系统生成后续订单或参与权益记录
超过 deadline 且人数不足则拼团失败
```

### 8.4 价格更新流程

```text
订单完成
系统写入 PriceRecord
系统计算商品参考价格
更新 Product.reference_price
系统更新相关 UserAsset.current_value
前端价格走势页面读取 PriceRecord
```

## 10. 数据一致性与事务设计

以下业务必须使用数据库事务：

| 场景      | 原因                          |
| ------- | --------------------------- |
| 创建订单    | 需要同时锁定挂单和资产                 |
| 支付成功    | 需要同时更新订单、支付、资产、价格、信用        |
| 完成换物    | 需要同时交换双方资产归属                |
| 参与拼团    | 需要控制 current_count，防止超出目标人数 |
| 管理员冻结用户 | 需要同步限制相关交易行为并记录日志           |

关键并发场景建议使用数据库行锁。Redis 可用于拼团、下单等高频操作的短期锁定，但最终一致性应以数据库事务为准。

## 11. 后端开发顺序建议

| 阶段   | 开发内容                       | 目标                |
| ---- | -------------------------- | ----------------- |
| 第一阶段 | common, accounts, products | 完成基础框架、登录认证、商品档案  |
| 第二阶段 | assets, market             | 完成资产管理和交易市场挂单     |
| 第三阶段 | orders, pricing            | 完成下单、模拟支付、价格记录    |
| 第四阶段 | exchanges, teams           | 完成换物和拼团核心流程       |
| 第五阶段 | credits, operations        | 完成信用评价、日志和后台监管    |
| 第六阶段 | 联调与测试                      | 与前端页面对接，修正接口和状态流转 |

第九讲 PPT 中提到，系统实施可以先实现系统轮廓或框架，再不断添加功能，逐步完善到完整系统，同时面向对象方法适合按层划分构件并先完成构件接口，以便并行开发。 因此后端开发应先统一模型、权限、接口规范，再分模块推进。

## 12. 设计调整说明

与原设计报告相比，后端实现时建议做以下调整：

| 原设计             | 调整建议                       | 原因                      |
| --------------- | -------------------------- | ----------------------- |
| 用户表和管理员表分开      | 合并为 User，通过 role 区分        | 减少重复认证逻辑，便于权限控制         |
| 订单直接关联商品        | 增加 Listing，订单关联挂单          | 交易市场需要表达卖家、资产、挂单价格和挂单状态 |
| 换物请求只有一张表       | 增加 ExchangeMatch           | 支持多人提交换物匹配，记录申请方资产      |
| 资产编号为 AS + 用户ID | 改为全局唯一资产编号                 | 一个用户可能拥有多个资产            |
| 商品编号包含卖家 ID     | 商品档案编号不绑定卖家，卖家关系放入 Listing | 商品是标准档案，出售行为才属于某个卖家     |

## 13. 总体结论

后端整体采用统一用户模型和 role 权限体系，不再把普通用户和管理员做成两套系统。系统模块按照基础数据、核心业务、决策分析、系统支撑四类展开，并落实为 accounts、products、assets、market、orders、exchanges、teams、pricing、credits 和 operations 等 Django App。

核心业务上，Product 表示平台商品档案，UserAsset 表示用户实际持有资产，Listing 表示出售挂单，Order 表示成交订单，ExchangeRequest 和 ExchangeMatch 表示换物流程，TeamProject 和 TeamParticipant 表示拼团流程。这样既能贴合现有设计报告，又能支撑前端页面和后续协同开发。
